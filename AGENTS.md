# AGENTS.md — Invosit (monorepo)

Context for code review agents (Codex) working on this repository. Bias reviews
toward what matters here, not generic Go best practices.

`CLAUDE.md` is the long-form reference for architecture, schema, routes, and
rationale. Read it when a section below needs more depth.

---

## What this repo is

Invosit syncs encrypted files alongside a git repo — for files that shouldn't be committed.

| Component | Location | Status |
|---|---|---|
| API server (Go) | `api/` (own `go.mod`) | shipping |
| CLI (Go) | `cli/` | not yet built |
| Frontend (web dashboard) | `frontend/` | not yet built |
| Docs | `docs/` | shared by all clients |

Each component is self-contained under its own top-level directory. **Security
is the core value proposition** — review every change through a security lens
first. When in doubt, prefer the more secure option even if it adds complexity.

The API is **outside the encryption boundary**. It must never see plaintext
file content or unwrapped DEKs. Anything that breaks this property is a blocker.
The frontend sits in the same trust position as the API — it must also never
handle plaintext file content.

## Tech stack (API)

- **Go 1.26+**, module `github.com/kyenel64/invosit/api` (lives at `api/go.mod`)
- HTTP: **stdlib `net/http` only** — `http.ServeMux` with 1.22 method+path patterns. No Gin, no Chi, no third-party router.
- Validation: `go-playground/validator/v10`
- Auth: **Ory Kratos** (separate container), validated via `/sessions/whoami`
- DB: Postgres via `database/sql` + `lib/pq`. No ORM.
- Storage: pluggable S3 / Cloudflare R2 behind `api/internal/storage.Storage`. CLI uploads/downloads via signed URLs — bytes never pass through the API.
- Tests: stdlib `testing`, `DATA-DOG/go-sqlmock` for DB.

## Commands

Go commands run from inside `api/` (that's where `go.mod` lives):

```bash
cd api
go build ./...
go vet ./...
go test -race -timeout 5m ./...
golangci-lint run --timeout=5m   # config: api/.golangci.yml
```

CI runs all four on every PR (with `working-directory: api`), plus
`govulncheck` and `gitleaks`. A PR that fails these locally won't pass
review.

## Working agreements

- **Don't introduce dependencies casually.** Stdlib covers most needs. Flag any new direct dependency in `go.mod` and check that it's actually necessary.
- **No global state.** API dependencies are injected via the `Handler` struct in `api/internal/handler/handler.go`. Flag package-level vars that hold runtime state.
- **Request-scoped data lives in `r.Context()`**, accessed via helpers in `api/internal/httpx`. Never read user IDs from request bodies or query params — middleware puts them in context after validating the session.
- **Migrations are forward-only**, numbered SQL files in `api/migrations/`. Flag column drops/renames without a deprecation step.
- **Timestamps UTC**, IDs are prefixed strings (`ws_`, `usr_`, `file_`, `ver_`, `grant_`, `log_`) generated by `api/internal/ids` — not UUIDs, not sequential integers.
- **Routes are registered in `api/internal/handler/routes.go`.** New endpoints belong there with the correct middleware wrapper.
- **OpenAPI is the contract.** `docs/openapi.yaml` is consumed by the (planned) CLI and frontend. Handler and openapi changes must move together.

## Security review checklist (API)

Apply this to every diff touching the API server. These rules are non-negotiable.

### Encryption boundary
- The API must never touch plaintext file content. Any handler that accepts or returns raw file bytes is wrong.
- Wrapped DEKs (`encrypted_dek` column, request bodies on file push, response bodies on file pull) must never be logged, cached, or inspected.
- Signed storage URLs are issued **after** auth + wrapped-DEK check, with max expiry **15 minutes** (`storage.MaxSignedURLExpiry`). Longer expiry returns `ErrExpiryTooLong` — don't soften this to a silent clamp.

### Authentication / authorization
- Every authenticated route must be wrapped by `middleware.RequireKratosSession`. Workspace routes must additionally be wrapped by `middleware.WorkspaceMember`. Order matters: session check first.
- File access requires **three layers**: session → workspace membership → wrapped DEK row. Missing any layer = 403.
- Trust user identity only from the request context (`httpx.UserID`-style accessors). Reject any handler that reads user IDs from request bodies/queries for authorization decisions.
- The Kratos webhook endpoint (`/internal/hooks/kratos/after-registration`) is gated by `X-Kratos-Webhook-Secret` via **constant-time comparison** (`subtle.ConstantTimeCompare`). Don't accept `==`.

### Input validation
- Request structs use `validator/v10` tags and are decoded via `httpx.Bind` (which sets `DisallowUnknownFields`). Flag handlers using `json.Decode` directly.
- File paths: reject `..`, absolute paths, null bytes, leading separators.
- Content hashes: 64-char lowercase hex only.
- File sizes: workspace quota must be enforced before accepting a push.
- Never interpolate user input into SQL — parameterised queries only (`$1`, `$2`, …).
- Never echo validator error text back to clients; map to a safe code/message.

### Information leakage
- Return **403, not 404**, when a user lacks access to an existing resource. 404 leaks existence. (Exception: workspace-scoped routes where membership is already confirmed by middleware.)
- Client-facing error messages must be generic. Full errors and stack traces go to server logs only.
- Don't include DB errors, internal paths, or validator messages in responses.

### Must never be logged
- Request bodies (may carry wrapped DEKs)
- `Authorization` / `Cookie` headers, Kratos session tokens
- `encrypted_dek` column values
- Storage credentials or signed URLs

Safe to log: method, path, status, duration, user ID, workspace ID, IP, request ID.

### HTTP hardening
- Security headers (`X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, HSTS, `Content-Security-Policy: default-src 'none'`, `Referrer-Policy: no-referrer`) are applied via middleware. Flag handlers that override them.
- Body limit is 10 MB via `middleware.BodyLimit` — flag increases without a clear reason.
- CORS in production must be a known-origins allowlist (`CORS_ALLOWED_ORIGINS`). Never `*`.

## Security review checklist (CLI)

The CLI doesn't exist yet, so most of this is forward-looking. When `cli/`
diffs start landing, apply these:

- **Encryption is the CLI's job.** AES-256-GCM with a fresh DEK per file. Never reuse a DEK across files or workspaces. Flag any code path that encrypts without a freshly generated DEK.
- **Private keys never leave the host.** Only the user's public key gets sent to the API. Flag any code that uploads, logs, or transmits private key material.
- **Wrapped DEKs are opaque.** The CLI unwraps DEKs locally using the user's private key. It must not log raw or wrapped DEK values.
- **Session tokens at rest** must be stored with appropriate filesystem permissions (e.g. `0600` on Unix). Flag world-readable token files.
- **Storage I/O is direct.** The CLI uploads/downloads encrypted blobs straight to the storage provider using signed URLs from the API. It must not proxy bytes through the API.
- **Manifest (`.invosit.yaml`) is plaintext metadata only.** Never write key material, session tokens, or wrapped DEKs into the manifest — it's committed to git.

This section grows when the CLI does. Add real flags as patterns emerge.

## Things to flag specifically

When reviewing a PR, push back on:

1. New third-party HTTP routers or web frameworks. Stdlib-only.
2. ORMs or query builders. `database/sql` directly.
3. `context.Background()` inside a request handler — propagate `r.Context()`. (`noctx` and `contextcheck` linters catch most; watch for new escapes.)
4. Unchecked errors from `db.Query`, `rows.Close`, `resp.Body.Close`.
5. SQL string concatenation with user input.
6. Returning `err.Error()` directly to clients. Wrap with `%w` server-side, return a safe message + stable code to clients.
7. New tests that don't use `sqlmock` when the existing patterns in `api/internal/handler/*_test.go` and `api/internal/middleware/*_test.go` do.
8. New env vars not documented in `.env.example` **and** `CLAUDE.md`.
9. `docs/openapi.yaml` and handler changes out of sync — they must move together. (Doubly important now that a CLI **and** a frontend will be generated/maintained off this contract.)
10. New routes registered outside `api/internal/handler/routes.go`, or registered without the right middleware wrapper.
11. CLI or frontend code reaching into `api/internal/` packages. The `internal/` rule blocks Go imports at compile time; cross-component HTTP coupling that bypasses `docs/openapi.yaml` is the moral equivalent and should be pushed back on.

## Out of scope for this repo

- File encryption from the **server's** point of view (CLI-side concern; the API must remain outside the encryption boundary).
- Plaintext file content on the frontend side (also outside the encryption boundary — same rule).
